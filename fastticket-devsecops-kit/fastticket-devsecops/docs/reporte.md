# Reporte de justificación (<=500 palabras)

Se propone un pipeline DevSecOps sobre **GitHub Actions** porque permite ejecutar jobs encadenados, runners hospedados por GitHub y protección por ambientes, suficiente para simular CI/CD empresarial sin infraestructura propia. El flujo cumple velocidad porque cada control es automático y paralelo donde conviene; y cumple seguridad porque toda promoción depende de gates que bloquean el avance ante fallos.

**A. Diseño de entornos.** En **Dev**, el desarrollador ejecuta pruebas unitarias, lint y validaciones locales con credenciales de prueba y variables `.env` no productivas. El hardening mínimo aquí es aislar secretos reales, limitar permisos y trabajar con datos ficticios. En **Staging/QA**, el workflow levanta `mongo + backend + frontend` con Docker Compose dentro del runner; no es producción real, pero sí un clon funcional suficiente para validar integración, smoke tests y DAST. En **Prod**, el despliegue se simula promoviendo el mismo bundle escaneado, más manifests Kubernetes endurecidos: contenedores non-root, `readOnlyRootFilesystem`, `seccompProfile`, eliminación de capacidades Linux, `NetworkPolicy`, secretos externos al código y aprobación manual antes de liberar.

**B. Herramientas.** Se elige **GitHub Actions** por simplicidad y porque todo el repositorio ya vive en GitHub. **Trivy** cubre la primera línea de defensa contra CVEs: analiza dependencias, filesystem, imágenes y también IaC. **Semgrep** es adecuado para Node.js/JavaScript porque encuentra patrones inseguros rápido y encaja bien con shift-left. **OWASP ZAP Baseline** se usa en staging porque DAST necesita una aplicación viva; allí puede revisar headers, cookies, exposición de rutas y configuraciones inseguras sin atacar producción. Para hardening, los **Dockerfiles multi-stage** y los manifests Kubernetes endurecidos reducen la superficie de ataque antes del release.

**C. Pipeline y gates.** 1) **CODE**: unit tests, lint y build. Gate: cobertura mínima 80%. 2) **BUILD & SCA**: construcción de imágenes y Trivy; gate: cero CVE High/Critical. 3) **SAST**: Semgrep; gate: cero hallazgos High. 4) **DEPLOY STAGING**: Compose en CI; gate: salud de API y frontend. 5) **TEST & DAST**: smoke tests + ZAP; gate: cero fallos funcionales y cero alertas DAST Medium/High. 6) **HARDENING FINAL**: validación de Kubernetes y políticas; gate: manifests seguros. 7) **DEPLOY PROD**: promoción simulada del artefacto ya aprobado.

**¿Por qué SAST antes que DAST?** Porque SAST detecta errores temprano, más barato y sin depender de un entorno desplegado: aplica el principio shift-left. **¿Cómo se logra seguridad y velocidad?** Automatizando controles, usando imágenes inmutables y evitando revisiones manuales excepto la última aprobación. **¿Qué protege más el hardening?** Secretos, tokens JWT, cadena de suministro de imágenes, acceso de red entre servicios y datos sensibles del flujo de compra.
